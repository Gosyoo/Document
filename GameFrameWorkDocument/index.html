1. 项目简介

GameFramework 是一个使用 C# 实现的、与具体游戏引擎解耦的游戏框架库（核心库），通常作为 Unity 工程中的核心层使用（配合上层适配层 UnityGameFramework）。它提供了游戏开发中最常用的功能模块，把复杂的资源、UI、网络、流程、事件等问题用一套统一的"模块化 + 接口驱动 + Helper 适配"模式解决。

项目特征

  • 引擎无关：核心库不依赖 UnityEngine，所有引擎相关操作都通过 IXxxHelper 接口下沉到上层。
  • 接口驱动：每个模块对外暴露 IXxxManager 接口，内部实现 internal sealed class XxxManager，外部只能通过接口访问。
  • 统一生命周期：所有模块继承自 GameFrameworkModule，由 GameFrameworkEntry 统一管理 注册 → 轮询 Update → 关闭 Shutdown。
  • 性能至上：内置引用池(ReferencePool)、对象池(ObjectPool)、任务池(TaskPool)、事件池(EventPool)等多套池化机制，最大限度降低 GC。
  • 状态驱动：游戏流程由 Procedure（基于 Fsm）驱动，结构清晰、可测可换。

2.3 接口与实现的设计模式

所有模块都遵循同一种"对外接口 + 内部实现 + Helper 适配"的三件套模式：

classDiagram class IXxxManager { <> +Method1() +Method2() } class XxxManager { <> -m_Helper : IXxxHelper +SetHelper(IXxxHelper) } class GameFrameworkModule { <> +Priority : int +Update() +Shutdown() } class IXxxHelper { <> } class UnityXxxHelper { "Unity 工程中的实现" } GameFrameworkModule <|-- XxxManager IXxxManager <|.. XxxManager XxxManager o--> IXxxHelper : 通过 SetHelper 注入 IXxxHelper <|.. UnityXxxHelper : 上层实现
💡 这种设计的好处：
1) 业务代码只面向 IXxxManager 接口，无法误用内部 API；
2) 引擎相关行为通过 IXxxHelper 注入，便于替换引擎或单元测试；
3) 所有 Manager 通过 GameFrameworkEntry.GetModule<T>() 反射 + 缓存获取。

3. 模块优先级与生命周期

每个模块都有一个 Priority 整数：优先级越高的模块越先 Update，越晚 Shutdown。这保证了关键依赖的初始化顺序与销毁顺序合理。下表是源码中各模块的实际优先级：

优先级模块说明
7EventManager事件分发要最先处理
6ObjectPoolManager对象池服务于 Entity/UI/Sound
5DownloadManager下载任务驱动
4FileSystemManager虚拟文件系统
3ResourceManager资源管理
2SceneManager场景
1FsmManager状态机轮询
0其他默认模块如 Config、DataTable、Localization、UI、Sound、Entity、Network、WebRequest、Setting、Variable…
-1DebuggerManager调试器最后渲染
-2ProcedureManager流程在最后驱动（避免依赖未就绪）
📌 GameFrameworkEntry.CreateModule 会按优先级降序插入到 LinkedList，Update 顺序遍历=高优先先 Update；Shutdown 反向遍历=高优先后 Shutdown。

4. 核心机制详解

4.1 GameFrameworkEntry：模块系统

GameFrameworkEntry 是一个静态类（位于 Base/GameFrameworkEntry.cs），它做三件事：

  1. 注册并缓存所有 GameFrameworkModule 实例（按 Priority 排序）；
  2. 每帧统一驱动 Update；
  3. 退出时统一 Shutdown。

关键代码（已简化）：

public static T GetModule<T>() where T : class
{
    Type interfaceType = typeof(T);            // 比如 IEventManager
    if (!interfaceType.IsInterface) throw ...; // 必须传入接口
    // 把 IXxxManager 转换成 XxxManager
    string moduleName = $"{interfaceType.Namespace}.{interfaceType.Name.Substring(1)}";
    Type moduleType = Type.GetType(moduleName);
    return GetModule(moduleType) as T;         // 找不到就反射创建
}
💡 命名约定：接口必须以 I 开头，去掉 I 后就是同 namespace 下的实现类名。例如 GameFramework.Event.IEventManager → GameFramework.Event.EventManager。

4.2 引用池 ReferencePool

引用池负责缓存可重用对象的实例，实现接口的对象会被自动缓存复用。它解决频繁 new/GC 的问题，是事件参数、任务、节点等高频对象的核心降耗手段。

使用方式

// 1. 让你的类实现 IReference 接口
public class MyEvent : GameEventArgs, IReference
{
    public int EventId { get; private set; }
    public static MyEvent Create(int id)
    {
        var e = ReferencePool.Acquire<MyEvent>(); // 拿出一个
        e.EventId = id;
        return e;
    }
    public void Clear()                       // 必须实现 Clear
    {
        EventId = 0;
    }
}

// 2. 用完归还
ReferencePool.Release(myEvent);

内部结构

classDiagram class ReferencePool { -Dictionary~Type, ReferenceCollection~ s_ReferenceCollections +Acquire~T~() +Release(IReference) +Add~T~(int count) +Remove~T~(int count) +ClearAll() } class ReferenceCollection { -Queue m_References +Acquire() +Release(IReference) } class IReference { <> +Clear() } ReferencePool *-- ReferenceCollection ReferenceCollection ..> IReference

4.3 事件池 EventPool

事件池是事件机制的核心容器，EventManager、SoundManager 内部都使用了它。它支持：

  • 线程安全发送：Fire 把事件入队，下一帧主线程统一分发；
  • 立即发送：FireNow 当前线程立刻分发；
  • 多种模式：EventPoolMode.AllowNoHandler / AllowMultiHandler / AllowDuplicateHandler。
sequenceDiagram participant Biz as 业务代码 participant Pool as EventPool participant Q as 事件队列 participant Main as 主线程 Update Biz->>Pool: Fire(sender, args) Pool->>Q: enqueue (锁保护) loop 每帧 Main->>Pool: Update(...) Pool->>Q: dequeue all Pool-->>Biz: 主线程回调 EventHandler end Note over Biz,Pool: FireNow 则跳过队列，直接回调

4.4 任务池 TaskPool<T>

任务池是资源加载、下载、网络请求等异步任务的核心调度器。它把任务（TaskBase）派给空闲的代理（ITaskAgent）执行：

关键类：TaskBase / TaskInfo / TaskPool / ITaskAgent / StartTaskStatus / TaskStatus

4.5 有限状态机 FSM

FSM 是 GameFramework 中所有"流程类"逻辑的基石，Procedure 模块就是基于它实现的。每个状态是一个继承自 FsmState<T> 的类：

public class ProcedureLogin : FsmState<IProcedureManager>
{
    protected internal override void OnEnter(IFsm<IProcedureManager> fsm) { ... }
    protected internal override void OnUpdate(IFsm<IProcedureManager> fsm,
        float elapseSeconds, float realElapseSeconds) { ... }
    protected internal override void OnLeave(IFsm<IProcedureManager> fsm, bool isShutdown){ ... }
}

// 切换状态
ChangeState<ProcedureMain>(fsm);

5.1 Resource 资源模块（最核心、最复杂）

这是整个框架最庞大的模块（ResourceManager.cs 超过 100KB，含 30+ 子文件），它要完成：

  1. 三种资源模式：Package(单机)、Updatable(预下载更新)、UpdatableWhilePlaying(边玩边下)。
  2. 版本管理：PackageVersionList/UpdatableVersionList/LocalVersionList/ResourcePackVersionList。
  3. 核心子组件：
    ResourceIniter（首次初始化）→ VersionListProcessor（更新版本表）→ ResourceChecker（对比本地远端）→ ResourceUpdater（下载差异）→ ResourceVerifier（CRC 校验）→ ResourceLoader（加载到内存）。
  4. 异步加载：通过 TaskPool<LoadResourceTaskBase> + ILoadResourceAgentHelper 调度。

5.2 ObjectPool 对象池

对象池是 Entity / UI / Sound 三大模块的"内功心法"，它的核心特性：

  • 支持多种对象类型，每种类型一个 ObjectPool<T>。
  • 每个对象（ObjectBase）有：名称、目标对象 Target、是否锁定、优先级、最近使用时间。
  • 释放策略：自动释放间隔(AutoReleaseInterval)、容量(Capacity)、过期时间(ExpireTime)、自定义过滤(ReleaseObjectFilterCallback)。
  • API：Spawn / Unspawn / Register / SetLocked / Release。
public abstract class ObjectBase : IReference
{
    public string Name; public object Target;
    public bool Locked; public int Priority; public DateTime LastUseTime;
    protected internal virtual void OnSpawn() { }
    protected internal virtual void OnUnspawn() { }
    protected internal abstract void Release(bool isShutdown);
}

5.3 Procedure 流程

Procedure 模块本身代码量很少 —— 它就是一个"用 FSM 包装的流程驱动器"。

// 1. 流程基类
public abstract class ProcedureBase : FsmState<IProcedureManager> { }

// 2. 初始化流程管理器
procedureMgr.Initialize(fsmManager,
    new ProcedureLaunch(),
    new ProcedureCheckVersion(),
    new ProcedureUpdateResources(),
    new ProcedureLogin(),
    new ProcedureMain());

// 3. 启动
procedureMgr.StartProcedure<ProcedureLaunch>();
💡 一个典型项目通常有 8~15 个 Procedure，分别对应：启动 → 初始化 → 检查版本 → 更新资源 → 预加载 → 登录 → 主城 → 战斗 → ……

5.4 Fsm 状态机

支持给任意类型对象（如 怪物、玩家、流程）创建状态机：

IFsm<Monster> fsm = fsmMgr.CreateFsm<Monster>("MonsterAI", monster,
    new IdleState(), new ChaseState(), new AttackState());
fsm.Start<IdleState>();

关键能力：状态切换、共享数据 SetData/GetData、状态间转移由状态自身触发 ChangeState<T>()。

5.5 Event 事件

事件用整数 ID 区分，事件参数继承自 GameEventArgs 并实现 IReference：

// 订阅
EventManager.Subscribe(MyEventArgs.EventId, OnMyEvent);
// 抛出（线程安全，下一帧分发）
EventManager.Fire(this, MyEventArgs.Create(data));
// 立即抛出
EventManager.FireNow(this, MyEventArgs.Create(data));

5.6 Entity 实体

Entity 用于管理"游戏世界中可显隐"的对象（怪物、子弹、特效、玩家形象等）。

  • EntityGroup：实体分组（如 Monster/Bullet/Effect），每组一个对象池。
  • ShowEntity / HideEntity：异步显示/隐藏，伴随事件参数。
  • 支持依赖资源加载：和 Resource 模块联动。

5.7 UI 界面

UIManager 维护 UIGroup 列表（如 Background/Default/HUD/Top/Tooltip），每个 Group 中维护多个 UIForm：

flowchart TD UIM[UIManager] UIM-->BG[UIGroup: Background
Depth=0] UIM-->DEF[UIGroup: Default
Depth=10] UIM-->HUD[UIGroup: HUD
Depth=20] UIM-->TOP[UIGroup: Top
Depth=100] DEF-->F1[UIForm: MainMenu] DEF-->F2[UIForm: SettingsForm] HUD-->F3[UIForm: HpBar] TOP-->F4[UIForm: Loading]

API：OpenUIForm、CloseUIForm、RefocusUIForm，每个操作都有对应的成功/失败/更新事件参数。

5.8 Scene 场景

异步 LoadScene/UnloadScene，事件参数：LoadSceneSuccess/Failure/Update/DependencyAsset，UnloadSceneSuccess/Failure。

5.9 Sound 声音

SoundGroup 管理 SoundAgent（具体的播放代理），支持优先级、淡入淡出、3D 参数 PlaySoundParams。

5.10 Network 网络

支持多通道(NetworkChannel)、TCP 同/异步，提供 Packet/PacketHeader 抽象，自动处理粘包：

flowchart LR App[业务] -->|Send| Ch[NetworkChannel] Ch -->|序列化| SS[SendState] SS --> Sock[(Socket)] Sock --> RS[ReceiveState] RS -->|解 PacketHeader| PH[IPacketHandler] PH --> App Ch -.-> HB[HeartBeatState 心跳]

5.11 Download 下载

由 DownloadManager + DownloadAgent + DownloadTask + DownloadCounter 组成。
- AddDownload 创建任务进队列；
- 多个 DownloadAgent 并发处理；
- DownloadCounter 统计实时速度（基于滑动窗口节点）；
- 事件：Start/Update/Success/Failure。

5.12 WebRequest 请求

结构与 Download 几乎一致：WebRequestManager + Agent + Task，主要做 GET/POST 接口请求。

5.13 Config 全局配置

解决"全局开关/数值"。提供 Bool/Int/Float/String 四类，支持从字节流/文本异步加载。
区别于 DataTable：DataTable 是结构化的"行表"，Config 是扁平的键值。

5.14 DataTable 数据表

对应 Excel 配置：DataTableManager.CreateDataTable<T>() 创建一张表，T 是实现 IDataRow 的行类型。常配合代码生成器使用。

5.15 DataNode 数据结点

类似带路径的 KV 仓库："Player.Bag.GoldCount"，支持 SetData/GetData，可以挂任意 Variable。

5.16 Localization 本地化

语言枚举 Language（涵盖几十种语言），GetString(key, args...) 支持参数化文本，配合资源系统按语言加载文本表。

5.17 Setting 玩家设置

负责玩家本地偏好持久化（如音量、语言、画质），上层一般以 PlayerPrefs/文件方式实现 ISettingHelper。

5.18 FileSystem 虚拟文件系统

这是相当独特的一个模块：把多个小文件打包到一个大文件里，运行时按"虚拟路径"读取。
组成：HeaderData(头) + 字符串区(StringData) + 块表(BlockData) + 数据区。
作用：减少 IO 数、加密保护、热更增量包。

5.19 Debugger 调试器

提供运行时调试窗口，按 DebuggerWindowGroup 树状组织 IDebuggerWindow。常见窗口：FPS、内存、对象池统计、引用池统计、设置查看器等。

7. 框架中使用的设计模式

模式体现
外观/服务定位器GameFrameworkEntry.GetModule<T>() 统一获取所有 Manager
桥接/适配器所有 IXxxHelper 把"引擎相关"行为下沉到上层适配层
对象池ReferencePool + ObjectPoolManager 双层池
状态模式Fsm，Procedure 是其特化
观察者/发布订阅EventManager + EventPool
命令模式TaskBase 抽象任务、由 Agent 统一执行
组合模式DataNode、DebuggerWindowGroup 树形结构
策略ResourceMode、EventPoolMode 等枚举切换不同策略
分部类(partial)大型 Manager 内含多个 partial 子类（如 ResourceManager 拆成 30+ 文件）

8. 推荐学习路径

🟢 第一阶段：理解骨架（约 2~3 小时）

  1. 读 Base/GameFrameworkEntry.cs + Base/GameFrameworkModule.cs 弄懂"模块如何被注册和驱动"。
  2. 读 Base/ReferencePool/ 整个目录，弄懂"为什么处处都要 IReference"。
  3. 读 Base/EventPool/，搞懂事件如何线程安全地分发。

🟡 第二阶段：理解三大支柱（约 3~5 小时）

  1. Fsm 模块：理解 FsmState<T> 的生命周期、ChangeState 工作方式。
  2. Procedure：发现它就是一个特化的 FSM。
  3. ObjectPool：理解 ObjectBase + Spawn/Unspawn/Release，自动释放策略。

🔵 第三阶段：理解资源管线（约 5~8 小时，难点）

  1. 先看 Base/TaskPool/ 整个目录，搞懂任务-代理调度模型。
  2. 再看 Resource/IResourceManager.cs 接口，了解全部能力。
  3. 按以下顺序串起子系统：
    VersionListProcessor → ResourceChecker → ResourceUpdater → ResourceVerifier → ResourceLoader
  4. 看 FileSystem 如何把多个小文件打成一个大包。

🟣 第四阶段：业务模块举一反三（按需）

  • UI、Entity、Scene、Sound：套路相同—— 都用 ObjectPool + Resource 异步加载 + 事件通知。看一个，其余即可类推。
  • Network：理解 PacketHeader → Packet → IPacketHandler 的处理链。
  • Download / WebRequest：结构与 ResourceLoader 极相似（任务+代理）。

🟤 第五阶段：实战

  1. 找一个 StarForce 这种官方示例 Unity 工程，搭起来跑起来。
  2. 新增一个自己的 Procedure（比如 ProcedureGuide），让它在 Login 之后插入。
  3. 新增一个自己的 EventArgs，验证 EventManager 流程。
  4. 修改一个 DataTable / Localization 试试热更替换。
🎯 学习心法：GameFramework 的所有模块几乎都遵循同一组"约定"——
① 接口对外、实现内部 sealed；
② 引擎相关下沉到 IXxxHelper；
③ 异步操作走 TaskPool/EventPool；
④ 高频对象走 ReferencePool/ObjectPool；
⑤ 流程类逻辑走 Fsm/Procedure。
看懂一个模块就等于看懂一类模块。
